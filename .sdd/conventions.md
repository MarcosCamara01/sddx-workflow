# Project Conventions

## Tech Stack

- Runtime: Node.js >= 18.
- Language: TypeScript in strict mode, compiled to CommonJS.
- CLI framework: `commander`.
- Interactive prompts: `@inquirer/prompts`.
- Bundler: `tsup`, outputting `dist/cli.js` with a `#!/usr/bin/env node` banner.
- Quality tooling: Biome for check, format, and lint.
- Tests: Node's built-in `node:test` runner plus `node:assert/strict`.
- Package surface: npm package `sddguard`; published files are limited to `dist/`, `templates/`, README, package metadata, and npm defaults.

## File & Folder Structure

```text
src/
  cli.ts
  providers.ts
  spec-state.ts
  utils.ts
  commands/
templates/
  workflow.md
  AGENTS.md
  CLAUDE.md
  gemini.md
  copilot-instructions.md
  claude-commands/
  codex-skills/
  copilot-prompts/
  gemini-commands/
  windsurf-workflows/
  cursor-rules/
  windsurf-rules/
  zed-rules/
  specs/_template/
  domains/
test/
smoke/
.github/workflows/ci.yml
```

Non-obvious:

- `dist/` is build output and is regenerated by `npm run build`.
- `templates/` is runtime package data; the built CLI reads it relative to `dist/cli.js`.
- `.sdd/`, `.agents/`, and `.claude/` in this repo are the repo's own dogfood install, not source templates.
- `specs/_template/` at repo root is dogfood context; `templates/specs/_template/` is the installable template source.

## Naming Conventions

- Source files use kebab-case where compound names are needed.
- Each subcommand module exports one named `<verb>Command` function.
- Provider IDs are lowercase and hyphenated where needed: `claude-code`, `codex`, `gemini`.
- Agent command names live in `src/commands/command-names.ts`.
- Spec folders in target projects use kebab-case under `specs/<name>/`.

## Code Style

- Keep the CLI procedural and small; do not introduce classes or a plugin architecture without an explicit design decision.
- Use synchronous `fs` operations unless a dependency requires async.
- Keep templates as copied data. Do not add templating syntax or runtime Markdown parsing inside `templates/`.
- Preserve the current console style: blank line, two-space indent, aligned action labels, and clear stderr errors.
- Treat `.sdd/project-overview.md`, `.sdd/conventions.md`, and `.sdd/domains/` as user-owned files.
- Treat provider entrypoints/rules (`AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, Copilot instructions, Cursor/Windsurf/Zed rules) as user-owned on `update` and `init --force`; command files may be refreshed.
- Prefer targeted changes over broad refactors; provider parity is a product requirement.

## Patterns to Avoid

- Editing only one command-aware provider template for a protocol change.
- Adding runtime enforcement to target projects without revisiting the installer-only architecture.
- Overwriting user-owned project context or provider entrypoints/rules from `update` or `init --force`.
- Treating plain `status` as a full verifier; use `status --strict` or `gate <phase> <feature>` for blocking checks.
- Adding dependencies when a small local helper is enough.
- Publishing changes that update README but not provider entrypoints or workflow templates.

## Testing

- Run `npm run check` for Biome checks.
- Run `npm test` for the full test suite; it also runs `npm run check` and `npm run build`.
- Run `npm pack --dry-run` before release-sensitive changes.
- Existing tests cover provider parity, package contents, install/update behavior, `doctor`, `commands`, `status`, `gate`, smoke flows, and an end-to-end installed workflow fixture.
- CI runs on Node 18, 20, and 22 for pushes to `main`/`master` and pull requests.

## Domain Glossary

- **SDD Protocol** — the Markdown workflow and command instructions installed into a project.
- **Provider** — an AI tool integration registered in `src/providers.ts`.
- **Command-aware provider** — a provider with one installed file per SDD command.
- **Rule-only provider** — a provider that gets always-on context instead of per-command files.
- **Core files** — `.sdd/workflow.md`, project context placeholders, and `specs/_template/`.
- **Workflow files** — managed files refreshed by `update`; user-owned provider entrypoints/rules are preserved.
- **Spec** — a feature folder under `specs/<name>/`.
- **Gap** — an implementation blocker recorded in `impl-gaps.md`.
- **CR** — a post-approval change request recorded in `amendments.md`.
- **Stop point** — a protocol point where the agent must halt for human input.
- **Gate** — an executable CLI check that fails non-zero when a phase has unresolved blockers.
